### Sprint A2: WebSocket PubSub (Week 2-3) — 14h

#### A2.1 — Redis PubSub for WebSocket Broadcast
**Effort**: 14h | **Priority**: P1 (required before scaling to 2+ workers)  
**What**: Replace in-process WebSocket broadcast with Redis PubSub so multiple uvicorn workers can serve the same WebSocket clients.  

**Sub-tasks**:

1. **Channel Design** (1h)
   - Channels: `ws:stock`, `ws:crypto`, `ws:forex`, `ws:all`
   - Signal gen task publishes to relevant channel: `redis.publish("ws:stock", signal_json)`

2. **PubSub Listener** (4h)
   - Background task in each uvicorn process subscribes to all channels
   - On message: decode JSON, fan out to local WebSocket connections
   - Per-client market filter applied locally (not at Redis level)

3. **Backpressure** (2h)
   - Per-client bounded message queue (max 50 messages)
   - Drop oldest if full, log warning
   - Client-side: if 5+ messages queued, show "X updates pending" badge

4. **Reconnection** (2h)
   - Redis PubSub reconnect with exponential backoff (1s → 2s → 4s → max 30s)
   - Log connection state changes
   - Health endpoint reports PubSub connection status

5. **Publisher Integration** (2h)
   - Update signal generation task to publish via Redis instead of direct broadcast
   - Update market data tasks to publish price updates
   - Serialize once in publisher (JSON bytes)

6. **Testing** (3h)
   - Unit: mock Redis, verify publish/subscribe
   - Integration: 2 workers, signal published on worker A, received on client connected to worker B
   - Load: 100 concurrent WebSocket clients, verify no message loss

**Acceptance**:
- [ ] Signals broadcast across multiple workers
- [ ] No message loss under normal load
- [ ] Graceful degradation if Redis PubSub disconnects
- [ ] Health endpoint reports PubSub status

### Sprint A3: AI/ML Improvements (Week 3-4) — 23h

#### A3.1 — Claude `tool_use` for Structured JSON
**Effort**: 2h | **Priority**: P1  
**What**: Use Claude's `tool_use` feature to get guaranteed JSON output instead of parsing free-text responses.  
**How**:
- Define tools with JSON schemas for sentiment analysis and signal reasoning
- Replace `json.loads(response.text)` with structured tool output
- Eliminates JSON parse failures entirely
- File: `backend/app/services/ai_engine/sentiment.py`, `reasoner.py`

---

#### A3.2 — Market-Specific Cache TTLs
**Effort**: 1h | **Priority**: P2  
**What**: Crypto news moves faster than forex. Cache sentiment differently per market.  
**How**:
- Crypto sentiment cache: 30 minutes (was 15 for all)
- Stock sentiment cache: 60 minutes
- Forex sentiment cache: 90 minutes
- File: `backend/app/services/ai_engine/sentiment.py`

---

#### A3.3 — Historical Calibration Curve
**Effort**: 4h | **Priority**: P1  
**What**: Map confidence scores to actual win probabilities using 90+ days of resolved signals.  
**How**:
- Bin resolved signals by confidence (0-20, 20-40, 40-60, 60-80, 80-100), minimum 50 per bucket
- For each bucket: actual_win_rate = hit_target / (hit_target + hit_stop)
- Apply isotonic regression to smooth the curve
- Display on dashboard: "Signals at 85% confidence historically hit target 72% of the time"
- File: `backend/app/services/signal_gen/calibration.py` (new)

---

#### A3.4 — News Semantic Dedup
**Effort**: 4h | **Priority**: P2  
**What**: Multiple news sources often report the same event. Deduplicate before sending to Claude.  
**How**:
- Use `sentence-transformers/all-MiniLM-L6-v2` to embed headlines
- Cosine similarity > 0.85 → same event, keep highest-quality source
- Reduces Claude API calls and prevents duplicate-inflated sentiment
- File: `backend/app/services/ai_engine/news_fetcher.py`

---

#### A3.5 — FinBERT vs Claude Evaluation
**Effort**: 8h | **Priority**: P3  
**What**: Evaluate whether FinBERT (free, deterministic, calibrated) can replace Claude for sentiment.  
**How**:
- Run FinBERT on same 500 articles Claude analyzed
- Compare: correlation, cost ($0 vs $15/mo), latency (local vs API), consistency
- If FinBERT scores within 0.1 correlation of Claude → switch for sentiment, keep Claude for reasoning only
- Decision doc, not full migration

---

#### A3.6 — Shadow Mode Comparison
**Effort**: 4h | **Priority**: P2  
**What**: Run old pipeline (v1.3 weights/logic) alongside new (v1.4) and compare signal quality.  
**How**:
- Fork scoring with feature flag: `SHADOW_MODE=true`
- Log both old and new scores to `shadow_comparison` table
- After 30 days: which pipeline had higher win rate?
- This proves v1.4 improvements actually helped
