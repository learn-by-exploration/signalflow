# SignalFlow AI — Multi-Persona Feature Review

> **25 reviewers × 5 disciplines** — identifying what to build next after v1.0.0
> Reviewed: 24 March 2026 | Current state: 350 frontend tests, 480+ backend tests, 9 pages, 38 components, 25+ API endpoints

---

## How This Review Works

Each reviewer evaluated the platform through their professional lens, asking:
1. **What's missing** for their user segment / discipline?
2. **What's the impact** (revenue, retention, trust, scalability)?
3. **What's the effort** (S/M/L/XL)?

Findings are consolidated into a prioritized feature roadmap at the end.

## Part 5: Architects (5 personas)

### A1 — Suresh (Backend Architect)
**Focus**: Scalability, performance, data pipeline

| # | Feature | Technical Impact | Priority |
|---|---------|-----------------|----------|
| A1.1 | **Event-driven architecture** | Replace polling-heavy Celery tasks with event streams (Redis Streams or Kafka). Data ingestion → analysis → signal should be a pipeline, not scheduled jobs | High |
| A1.2 | **Signal versioning** | When we change the scoring algorithm, we need to tag signals with the algorithm version. "This signal was generated by algo v2.1" | High |
| A1.3 | **Read replicas for API** | Dashboard reads should hit a read replica. Signal generation writes to primary. Separate read/write paths | Medium |
| A1.4 | **Data retention policy** | market_data table grows ~100K rows/day. Define 90-day hot, 1-year warm, archive cold. TimescaleDB compression policies | High |
| A1.5 | **Circuit breaker for data sources** | If yfinance API is down, circuit breaker should prevent retry storms. Currently just catches exceptions and retries | Medium |

| # | Feature | Technical Impact | Priority |
|---|---------|-----------------|----------|
