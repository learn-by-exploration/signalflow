### Sprint B — Portfolio Currency Fix (2-3 days)

The portfolio aggregation bug is real but the fix needs careful design. Two options:

#### Option A: Segregated display (recommended for v1.1)
- **Backend:** Change portfolio summary to return per-currency subtotals instead of a single aggregate. Add `currency` field inferred from market_type (stock→INR, crypto→USD).
- **Frontend:** Show separate P&L cards per currency. Remove misleading "Total Return %" that mixes currencies. Equity curve becomes per-market.
- **Allocation pie chart:** Show percentage within each market, or remove cross-market pie.
- **Size:** M (3-4 hours)
- **Risk:** Low — doesn't require exchange rates, forex API, or schema migration

#### Option B: Base currency conversion (deferred to v1.2)
- Add `base_currency` preference to user settings
- Fetch live USD/INR rate from forex data already ingested
- Convert all positions to base currency before aggregating
- Requires: forex rate lookup in portfolio endpoint, user preference storage
- **Size:** L (6-8 hours)

**Decision:** Ship Option A in v1.1. Option B in v1.2 after we have user feedback on whether cross-market aggregation is actually wanted.

#### B1. Portfolio summary — per-currency breakdown
- **File:** `backend/app/api/portfolio.py`
- **Change:** Group positions by currency (inferred from market_type). Return `{ by_currency: { INR: {invested, value, pnl, pnl_pct}, USD: {...} } }` instead of flat totals.
- **Schema:** Update `PortfolioSummary` in `backend/app/schemas/p3.py`

#### B2. Frontend — per-market P&L display
- **File:** `frontend/src/app/portfolio/page.tsx`
- **Change:** Render separate P&L cards per currency. Remove single "Total Return" that mixes currencies. Keep individual position cards (already correct).

#### B3. Equity curve — per-market or remove
- **File:** `frontend/src/app/portfolio/page.tsx`
- **Change:** Either split equity curve by market_type or hide it if user trades in multiple currencies.

#### B4. Allocation pie chart — fix percentages
- **File:** `frontend/src/components/charts/AllocationPieChart.tsx`
- **Change:** Show allocation within each market, not cross-market percentages.

### Sprint C — Position Sizing Calculator (2 days)

Upgrade the existing RiskCalculator from "what's my P&L at ₹X?" to "how much should I invest?"

#### C1. Add position sizing mode
- **File:** `frontend/src/components/signals/RiskCalculator.tsx`
- **Change:** Add a toggle: "Calculate P&L" (current) vs "Calculate Position Size" (new). In position sizing mode:
  - Input: account equity (₹ or $), risk tolerance (1%, 2%, 5%)
  - Output: recommended position size = (account × risk%) / (entry - stop_loss)
  - Show: "Risk ₹X to potentially gain ₹Y" in plain language
- **Size:** M (4-5 hours)
- **Risk:** Low — client-side calculation only
- **Tests:** Unit tests for position size formula

### Sprint D — Engagement & Retention (3-4 days)

#### D1. "Did you take this trade?" button
- **Backend:** Add `user_action` field to Signal response or create `SignalFeedback` model with `{signal_id, action: 'took'|'skipped'|'watching', entry_price?, notes?}`
- **Frontend:** Add button on SignalCard (collapsed) and signal detail page. Capture user's actual entry price for later performance tracking.
- **Size:** M (4-5 hours)
- **Value:** Highest-leverage engagement feature. Links signals to real user behavior. Enables "Your Track Record" page later.

#### D2. Free tier Telegram morning brief
- **Backend:** Check if morning brief task in `alert_tasks.py` filters by tier. If so, remove tier gate for morning brief specifically — it's the cheapest possible retention hook.
- **Size:** XS-S (1-2 hours to verify and ungate)
- **Value:** Zero-cost push engagement for free users. Without this, free users get zero proactive value.

### Sprint E — Market Safety (3-4 days)

#### E1. Central bank event suppression for forex
- **Design:** Before generating forex signals, check if a major central bank event (FOMC, RBI MPC, ECB, BOJ) is within ±4 hours.
- **Implementation:** Add economic event calendar (static JSON of 2026 dates initially, API-driven later). Signal generator skips forex pairs affected by upcoming events, or caps confidence at 40 (HOLD).
- **Size:** M-L (5-6 hours)
- **Risk:** Medium — need to define "which events affect which pairs" mapping
- **Value:** Prevents the most irresponsible signal scenario (BUY EUR/USD 2 hours before FOMC)

#### E2. Earnings calendar integration for stocks
- **Design:** Before generating stock signals, check if the symbol has earnings within ±2 trading days.
- **Implementation:** EventCalendar model already exists. Need earnings date fetcher (scrape moneycontrol.com or use Yahoo Finance earnings calendar). Generator suppresses or warns.
- **Size:** L (6-8 hours — needs data source research)
- **Risk:** Medium — earnings date accuracy varies by source
- **Value:** Prevents the most dangerous stock signal scenario (BUY before bad quarterly results)

|------|--------|------------|
